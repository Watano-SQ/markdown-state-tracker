# State Output Quality Spec

## Goal

在不扩大当前仓库边界的前提下，为第一阶段“输入边界与来源类型”改进冻结一组最小、可验证的实现边界。

这一阶段不试图直接解决全部输出质量问题，而是先回答以下基础问题：

- `input_docs/` 下哪些文件应进入状态追踪主链路，哪些不应进入
- 单篇 Markdown 内哪些内容属于作者自述，哪些属于元数据、引用材料、外部资料或结构化噪声
- 不同来源类型默认应如何处理：进入抽取、仅作为上下文、还是直接跳过
- 哪些切分边界不应被当前 chunking 继续跨越

任务完成后，应满足以下事实：

- 后续实现者无需再重新澄清“什么算输入样本”“什么算作者状态候选来源”
- 输入边界问题与主体规范化、候选过滤、显著性排序这些后续阶段问题被明确拆开
- `文档切片探索.md` 中可借鉴的思路会被限制在“结构先验 + 完整块保护 + 轻量规则”这一层，不被误用为当前仓库的直接蓝图

## Stage-1 Problem Focus

本阶段只聚焦以下两类问题：

### 1. 文件级输入边界过宽

当前扫描逻辑会处理 `input_docs/` 下所有 `*.md`。这会把规则文档、测试夹具或其他非用户内容样本一起纳入主链路，带来后续污染风险。

这一阶段需要冻结的不是具体实现细节，而是产品边界：

- 哪些文件属于“用户内容样本”
- 哪些文件属于“规则/测试/控制文件”
- 非样本文档应如何被显式排除，而不是依赖偶然未被抽取

### 2. 文档内部来源类型混杂

当前样本文档里同时存在：

- front matter
- 元数据表格
- 作者正文叙述
- 长段 blockquote 问答
- 教程式说明
- 抓取结果中的 HTML/JSON 片段
- 配置片段、代码片段、图片占位

但当前主链路仍主要把它们当作同质正文处理，再交给抽取层。结果是作者自述、引用说明、网页资料和结构化噪声一起进入状态候选。

本阶段要冻结的是：

- 这些块类型的最小分类法
- 各类型的默认处理策略
- 切分时不得跨越的结构边界

## Source Type Model

第一阶段实现应至少支持以下来源类型；命名可在实现时调整，但语义不可偏离。

### 1. 文档级元数据块

包括：

- 顶部 front matter
- 明显是标题、作者、创建时间、更新时间等文档元信息的区域

默认处理：

- 不作为正文 state extraction 输入
- 可提炼为文档上下文，例如作者名、文档标题、文档时间

### 2. 元数据表格块

包括：

- “作者 / 创建时间 / 更新时间”这类记录性表格
- 明显用于展示文档属性而非表达作者状态的表格

默认处理：

- 不直接进入状态抽取正文
- 如确有字段价值，应通过轻量解析进入文档上下文，而不是保留为普通 chunk 文本

### 3. 作者正文叙述块

包括：

- 第一人称复盘
- 个人计划、问题、尝试、结果、反思
- 明显由文档作者直接陈述的经验性内容

默认处理：

- 这是状态抽取的主来源
- 后续阶段如需主体规范化，应主要建立在这类块上

### 4. 引用问答 / 引用材料块

包括：

- 连续 blockquote
- `> Q:` / `> A:` 样式问答
- 明显引用他人、模型或外部来源的整段材料

默认处理：

- 保持为独立来源块，不与作者正文合并
- 默认不直接作为个人状态抽取来源
- 如后续需要使用，只能作为支持性上下文，而不是直接升格为作者状态

### 5. 外部资料 / 教程说明块

包括：

- 通用概念解释
- 面向“用户/您”的教程建议
- 抓取到的网页描述、说明性文字、公告摘要

默认处理：

- 与作者正文分开
- 默认不直接作为个人 `state_candidate` 来源
- 若作者在正文中明确采纳、实践、评价或复盘，应以后者为主，而不是以前者为主

### 6. 结构化转储 / 配置 / 代码块

包括：

- JSON / HTML 片段
- 配置对象
- fenced code block
- 大段命令或程序输出

默认处理：

- 默认不进入状态抽取正文
- 如确需保留，应以“结构化参考块”身份单独存在，而不是与自然语言叙述合并

### 7. 媒体占位与空信息块

包括：

- 单独图片链接
- 仅附件链接、空标题、空段落

默认处理：

- 默认跳过
- 不应单独形成状态抽取 chunk

## Boundary Rules

第一阶段实现应冻结以下边界规则：

- front matter 必须先于普通正文被识别，不得与正文合并成同一个抽取块
- 连续 blockquote 必须保持为独立块，不得与前后作者正文合并
- 元数据表格必须与作者正文分离
- fenced code block、HTML/JSON 转储块必须与自然语言叙述分离
- 图片占位、附件链接和纯空白区不应驱动新的状态候选
- 标题层级可作为辅助边界，但不能替代来源类型识别
- 小块合并只能发生在来源类型判断之后，不能先混合再标型

## File-Level Inclusion Boundary

第一阶段实现需要为 `input_docs/` 明确一条文件级纳入规则。

至少应满足：

- 规则文档不应作为用户状态样本进入主链路
- 测试夹具或纯验证文件不应默认参与正式状态输出
- “是否属于样本输入”应由显式规则决定，而不是依赖数据库残留状态或恰好未抽取

以下细节目前仍可能需要人类确认：

- 是否采用排除名单、命名约定、子目录约定，还是显式 include 机制
- `test_*.md` 是否全部视为测试夹具，还是允许其中一部分作为合成样本参与验证

若无法从仓库现状可靠推导，应标为 `需要人类确认`。

## Chunking Exploration Assessment

参考文件：[文档切片探索.md](/D:/Apps/Python/lab/personal_prompt/文档切片探索.md)

这份文档对第一阶段**有启发，但用途必须收窄**。

可借鉴：

- 结构先验优先于纯 token 切分
- 把问答块、定义块、教程块视为语义完整块
- 在识别来源类型之后再决定小块合并

本阶段不直接引入：

- LLM 主导切片
- 面向教材/参考书的层级索引
- `pdf -> Reference/Skill` 风格的重型流水线

原因：

- 当前仓库目标仍是“从 Markdown 生成状态文档”
- 第一阶段要解决的是输入边界，不是建立新的知识索引产品

## Non-goals

本阶段明确不解决以下问题：

- 不解决主体统一口径（作者 / 用户 / 我 / 人名）的最终表达规则
- 不解决 `state_candidates` 的作者相关性过滤规则
- 不解决显著性排序、输出收敛或最终展示策略
- 不解决语义去重、canonical summary 或主题归并
- 不修复 `documents.status` 的完整状态机与失败 chunk 恢复
- 不引入向量检索、embedding、外部知识库或多阶段文档理解流水线
- 不默认新增重型 schema

## In Scope

- 冻结 `input_docs/` 的文件级纳入边界
- 冻结单篇文档内最小来源类型分类法
- 为每种来源类型定义默认处理策略
- 冻结当前 chunking 不应跨越的结构边界
- 明确第一阶段实现应覆盖的最小验证面
- 记录仍需人类确认的边界问题

## Out Of Scope

- 当前 turn 内直接修改输入层、抽取层、聚合层或输出层代码
- 定义第二阶段的主体规范化细则
- 定义第三阶段的显著性、去重和输出排序细则
- 声称上述边界已经在仓库中实现

## Constraints

- keep current repository structure unless intentionally changed
- do not assume behavior that is not implemented
- mark uncertain facts as `需要人类确认`
- 保持当前仓库是本地 Markdown 状态追踪原型，而不是通用文档理解平台
- 优先采用轻量、可解释的结构规则，而不是新增复杂 LLM 编排
- `input_docs/` 视为潜在敏感输入，不在 retained docs 中复制长篇原文
- 若某类块是否应“跳过”还是“仅降权”无法从当前任务边界稳定推出，标为 `需要人类确认`

## Acceptance Criteria

- 存在一份独立 spec，且其范围只覆盖第一阶段“输入边界与来源类型”
- spec 明确区分文件级边界与文档内来源类型边界
- spec 至少覆盖以下来源类型：文档级元数据、元数据表格、作者正文、引用问答/引用材料、外部资料/教程说明、结构化转储/配置/代码、媒体占位/空信息块
- spec 为每种来源类型给出默认处理策略，而不是只列举问题现象
- spec 明确规定若干不可跨越的结构边界，避免继续按同质正文切块
- spec 明确限定 `文档切片探索.md` 的借鉴范围，只保留对第一阶段有价值的轻量思路
- 后续实现若完成本 spec，对当前样本文档应至少满足以下结果：
  - front matter 与元数据表格不会作为普通正文进入状态抽取
  - 长段 blockquote 问答不会与作者正文混成同一个抽取单元
  - HTML/JSON/配置转储不会默认直接进入状态抽取正文
  - 规则文档与测试夹具不会默认进入正式状态样本扫描
