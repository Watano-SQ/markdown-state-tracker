# Markdown State Tracker

一个基于本地 Markdown 文档的状态追踪与增量更新原型系统。

将不断积累的本地 md 文档，转化为一份可更新、可复用的状态文档。

## 🎯 项目目标

从目标文件夹中的本地 Markdown 文档中提取阶段性信息，组织成统一的"状态文档"，并在后续文档新增或修改时持续更新该输出。

**这是一个边界明确的原型，不是：**
- ❌ 通用聊天助手
- ❌ 完整长期记忆系统
- ❌ 开放式搜索系统
- ❌ 平台级产品

**而是：**
- ✅ 轻量级本地文档处理工具
- ✅ 可控的状态视图生成器
- ✅ 支持增量更新的原型系统

## 🏗️ 系统架构

三层 workflow 设计：

```
┌─────────────────────────────────────────────────────┐
│  输入层 (Input Layer)                                │
│  - 扫描目录中的 md 文件                              │
│  - 识别新增和修改过的文件                            │
│  - Chunk 切分（按段落/句子）                          │
│  - 结构化抽取（预留接口）                             │
└────────────────┬────────────────────────────────────┘
                 │
┌────────────────▼────────────────────────────────────┐
│  中间层 (Middle Layer) - SQLite                      │
│  - 存储文档、chunks、提取结果                         │
│  - 管理状态项（states）                               │
│  - 维护关系（relations）                              │
│  - 检索候选对象（retrieval_candidates）               │
└────────────────┬────────────────────────────────────┘
                 │
┌────────────────▼────────────────────────────────────┐
│  输出层 (Output Layer)                               │
│  - 按需从中间层选择状态项                             │
│  - 生成固定骨架的状态文档                             │
│  - 保存输出快照                                       │
└─────────────────────────────────────────────────────┘
```

## 📊 数据模型

### 核心表结构

1. **documents** - 原始文档信息
   - 路径、标题、修改时间、hash、处理状态

2. **chunks** - 文档切分后的片段
   - 关联文档、索引、文本、token 估算

3. **extractions** - 每个 chunk 的结构化提炼结果
   - JSON 格式存储抽取结果、模型版本

4. **states** - 聚合后的状态项
   - 分类（动态/静态）、子类型、摘要、详情、置信度

5. **relations** - 对象之间的关系
   - 源/目标类型和 ID、关系类型、置信度

6. **retrieval_candidates** - 待补充检索的候选对象
   - 表面形式、标准化名称、类型猜测、证据计数

7. **output_snapshots** - 输出文档的版本快照
   - 内容、生成时间、基于的状态版本

## 🚀 快速开始

### 环境要求

- Python 3.7+
- SQLite 3（Python 内置）

### 安装与运行

```bash
# 克隆项目
git clone https://github.com/yourusername/markdown-state-tracker.git
cd markdown-state-tracker

# 放入你的 Markdown 文档
# 将 .md 文件放到 input_docs/ 目录

# 运行处理流程
python main.py

# 查看生成的状态文档
cat output/status.md
```

### 命令行参数

```bash
# 强制重新初始化数据库
python main.py --init

# 查看统计信息
python main.py --stats

# 安静模式（不输出详细日志）
python main.py --quiet
```

## 🔧 配置

在 `config.py` 中可修改：

- `INPUT_DIR` - 输入文档目录
- `OUTPUT_FILE` - 输出文档路径
- `DB_PATH` - 数据库路径

在 `layers/output_layer.py` 中的 `OUTPUT_CONFIG` 可配置：

- 输出文档的分类和子类型
- 每个子类型的最大条目数

## 📝 输出文档结构

生成的状态文档采用固定骨架：

```markdown
# 状态文档

## 动态状态
### 进行中的项目
### 近期事件
### 待办事项
### 当前关注

## 稳定状态
### 偏好设定
### 背景信息
### 技能/能力
### 人际关系

## 归档区
```

- **动态状态**：近期活跃、正在进行的事项
- **稳定状态**：长期不变或缓慢变化的信息
- **归档区**：历史状态（通过数据库查询）

## 🎨 设计原则

### Token 控制

- **输入层**：按 chunk 分批处理，避免单次输入超长
- **输出层**：按需选取状态项，不把整个中间层塞给生成器

### 边界控制

- 输出不是无限增长的列表
- 通过归档机制和条目上限控制规模
- 主视图只保留 active 状态

### 可扩展性

- 抽取器接口预留，可替换为 LLM 或规则引擎
- 输出模板可自定义
- 中间层支持灵活查询

## 💡 设计理念

这个项目遵循"最小可行、易于验证"的原则：

1. **不过度工程化** - 优先使用 SQLite 而非重型数据库
2. **模块清晰** - 三层分离，便于替换和扩展
3. **受控边界** - 明确的项目范围，不做平台级功能
4. **可调试性** - JSON 存储原始抽取结果，便于检查
5. **渐进式** - 核心流程先跑通，再优化细节

---

**注意**：当前版本为最小可行原型，抽取器部分需要根据实际需求自行实现或集成 LLM。
